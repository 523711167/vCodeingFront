# React 内容运营后台技术方案文档

## 1. 文档概述

### 1.1 文档目标

本文档用于定义“内容/运营类后台管理系统”的前端工程技术方案，目标是让后续实现者可以直接依据本文档完成项目初始化、工程搭建、模块拆分与 MVP 开发。

本文档聚焦首版后台系统建设，强调以下几点：

- 以桌面端运营人员为主要使用对象。
- 提供统一且稳定的 React 工程架构。
- 明确登录、权限、路由、接口、状态管理等核心约束。
- 支持前端先基于 Mock 独立开发，再平滑切换真实接口。
- 为后续任务拆分、协作开发和持续迭代提供统一基线。

### 1.2 项目定位

该项目是一个桌面优先的后台管理 Web 应用，主要服务于内容团队与运营团队，用于支撑日常内容发布、运营活动配置、数据查看以及系统管理。

### 1.3 首版建设目标

首版目标限定为以下四个核心方向：

- 内容管理：支持内容列表查询、新增、编辑、状态管理、删除等基础能力。
- 运营活动：支持活动列表、活动配置、活动状态控制与基础运营操作。
- 基础数据看板：展示内容与运营相关的基础指标、趋势和快捷入口。
- 账号与权限管理：支持账号、角色、菜单权限和按钮权限配置。

### 1.4 暂不纳入范围

为了避免过度设计，首版明确不纳入以下能力：

- SSR 服务端渲染
- 微前端拆分
- 多语言国际化
- 复杂可视化大屏
- 低代码配置平台
- 拖拽式页面编排
- 面向移动端的专门适配方案

## 2. 建设原则

项目实施遵循以下工程原则：

- 简洁优先：先满足中后台业务稳定交付，再考虑复杂抽象。
- 约定优先：统一目录、命名、状态边界与接口模式，降低协作成本。
- 业务分域：页面与模块按照业务域拆分，而不是只按技术类型堆叠。
- 复用优先：优先复用通用布局、通用表格、通用表单能力，避免重复建设。
- Mock 先行：前端在后端接口未完成前即可独立开发和验证页面行为。
- 类型优先：首版默认使用 TypeScript，核心路由、接口返回、权限数据和全局状态都建立明确类型约束。

## 3. 技术选型

### 3.1 核心技术栈

首版工程基线固定为以下组合：

- `Vite`：构建工具与本地开发服务器
- `React`：UI 框架
- `TypeScript`：静态类型系统
- `React Router`：路由管理
- `Ant Design`：中后台组件库
- `Redux Toolkit`：全局状态管理
- `Axios`：HTTP 请求封装
- `ESLint + Prettier`：代码规范与格式统一

### 3.2 选型理由

#### Vite

- 冷启动和热更新速度快，适合前端日常开发。
- 配置简单，适合从零搭建 React 后台项目。
- 生态成熟，便于后续扩展环境变量、代理、构建优化等能力。

#### React

- 组件化能力成熟，适合后台系统高复用页面结构。
- 社区活跃，配套库稳定。
- 与 Ant Design、Redux Toolkit、React Router 配合成熟。

#### React Router

- 满足中后台常见的嵌套路由、守卫跳转、动态菜单映射需求。
- 易于结合权限模型实现动态路由过滤。

#### Ant Design

- 表格、表单、弹窗、布局等后台高频组件完整。
- 上手成本低，适合内容与运营后台快速交付。
- 能降低首版自研组件成本，将精力集中在业务实现。

#### Redux Toolkit

- 能以较低模板成本提供可维护的全局状态组织方式。
- 适合管理登录态、用户信息、权限数据、全局 UI 状态等共享状态。
- 比传统 Redux 更简洁，也更适合团队标准化协作。

### 3.3 辅助工程能力

项目默认补齐以下基础设施：

- `Axios` 请求实例与拦截器
- `tsconfig.json` 与路径别名配置
- `.env.development`、`.env.test`、`.env.production` 环境变量
- `ESLint` 代码检查
- `Prettier` 统一格式
- Mock 数据方案
- 构建与部署脚本

## 4. 工程架构设计

### 4.1 推荐目录结构

```text
src/
  app/
  router/
  layouts/
  pages/
  store/
  services/
  components/
  features/
  utils/
  mock/
  styles/
```

### 4.2 目录职责说明

#### `src/app`

应用级初始化逻辑，负责挂载路由、全局 Provider、全局样式、错误边界和应用入口组合。

#### `src/router`

维护静态路由、动态业务路由、路由守卫逻辑和菜单映射逻辑。

#### `src/layouts`

存放页面布局相关实现，至少包括：

- 登录布局
- 主框架布局
- 异常页布局

#### `src/pages`

页面级组件，按业务域组织，不按“list/detail/form”纯技术层组织。建议结构如下：

```text
pages/
  dashboard/
  content/
  operation/
  system/
  profile/
  exception/
  auth/
```

#### `src/store`

集中定义 Redux Toolkit 的 store、slice、middleware 和公共 selector。

#### `src/services`

按业务域维护接口请求方法，不直接在页面内书写裸 `axios` 请求。

#### `src/components`

沉淀跨页面复用组件，例如：

- 通用搜索栏
- 通用列表工具栏
- 通用状态标签
- 权限按钮组件
- 页面容器组件

#### `src/features`

承载可复用的业务能力模块，例如登录态恢复、权限过滤、菜单构建、分页查询封装等。

#### `src/utils`

通用工具方法，如本地缓存、日期处理、权限判断、查询参数格式化等。

#### `src/mock`

维护本地 Mock 数据、接口模拟逻辑与场景样例数据。

#### `src/styles`

全局样式、主题覆盖、变量文件和通用布局样式。

### 4.3 页面组织原则

页面以业务域组织，而不是按技术类型散落在各处。首版建议拆为以下模块：

- `dashboard`：首页看板
- `content`：内容管理
- `operation`：活动与运营管理
- `system`：账号、角色、菜单权限
- `profile`：个人中心
- `auth`：登录与鉴权相关页面
- `exception`：`403`、`404`、异常页

## 5. 路由设计

### 5.1 路由策略

路由采用“静态基础路由 + 登录后按权限过滤业务路由”的方案。

静态基础路由包含：

- 登录页
- 403
- 404
- 默认重定向

动态业务路由包含：

- 首页看板
- 内容管理
- 运营活动
- 系统管理
- 个人中心

### 5.2 路由元信息约定

每个业务路由必须具备统一元信息，用于菜单渲染、页面标题、权限判断与缓存策略控制。推荐定义如下：

```ts
interface AppRouteMeta {
  title: string;
  icon?: string;
  authCode: string;
  hidden?: boolean;
  keepAlive?: boolean;
}

interface AppRouteItem {
  path: string;
  element: React.ReactNode;
  children?: AppRouteItem[];
  meta: AppRouteMeta;
}
```

字段定义如下：

- `title`：菜单名和页面名展示文案
- `icon`：菜单图标标识
- `authCode`：页面访问权限码
- `hidden`：是否在侧边菜单中隐藏
- `keepAlive`：是否启用缓存保留页面状态

### 5.3 路由处理规则

- 未登录访问受保护页面时，统一跳转登录页。
- 登录成功后，根据用户权限拉取或组装可访问业务路由。
- 无权限访问页面时，统一跳转 `403`。
- 无匹配路由时，统一跳转 `404`。
- token 失效时，清空本地登录态和权限数据，并回到登录页。

## 6. 登录与权限模型

### 6.1 登录态方案

登录态以 token 为核心，持久化存储在浏览器本地存储中。应用刷新后从本地恢复登录信息，并重新拉取用户资料与权限信息。

首版流程定义如下：

1. 用户提交账号密码。
2. 登录成功后获取 token。
3. 本地持久化 token。
4. 拉取当前用户信息。
5. 拉取权限相关数据。
6. 构建菜单与动态路由。
7. 进入首页看板。

### 6.2 权限粒度

首版权限固定为三层：

- 页面路由权限：控制页面能否进入。
- 菜单可见权限：控制左侧导航是否展示。
- 按钮操作权限：控制新增、编辑、删除、发布、导出等按钮是否可见或可操作。

### 6.3 权限数据结构约定

前端默认后端返回以下信息：

- 用户基础信息
- 角色列表
- 菜单权限列表
- 按钮权限码列表

推荐返回结构如下：

```ts
interface PermissionMenu {
  path: string;
  title: string;
  authCode: string;
  children?: PermissionMenu[];
}

interface PermissionPayload {
  userId: string;
  name: string;
  roles: string[];
  menus: PermissionMenu[];
  buttons: string[];
}
```

### 6.4 权限映射规则

- 菜单权限通过 `path` 或 `authCode` 映射到前端路由配置。
- 页面是否可进入由路由级 `authCode` 判断。
- 按钮权限通过统一权限判断函数或权限按钮组件控制显隐。
- 若后端返回菜单树，则前端优先复用后端树结构；若只返回权限码，则前端基于本地路由配置过滤。

### 6.5 退出与失效处理

退出登录时必须统一清理：

- token
- 用户信息
- 权限信息
- 动态路由缓存
- 与当前账号相关的本地缓存数据

接口返回登录过期时，前端统一提示并回退登录页，避免页面停留在无效状态。

## 7. 状态管理设计

### 7.1 Store 划分

全局 store 至少拆分为以下三个领域：

- `auth`：登录状态、token、登录中状态
- `permission`：用户信息、角色、菜单、按钮权限、动态路由
- `app`：侧边栏折叠、主题设置、全局 loading、面包屑等 UI 级状态

### 7.2 状态边界原则

以下信息进入 Redux Toolkit：

- 登录状态
- 当前用户信息
- 权限码
- 菜单树
- 动态路由结果
- 应用级 UI 状态

以下信息不进入全局 store，优先保留在组件内部：

- 表单输入中间态
- 列表页临时筛选项
- 弹窗开关状态
- 单页局部 loading

### 7.3 状态管理约定

- 所有全局状态必须集中在 `store` 中维护，不允许页面私自复制全局权限数据。
- 页面读取全局状态时优先通过 selector 访问。
- 页面发起全局状态变更时统一通过 slice action 或 thunk 触发。
- 服务端数据以请求结果为准，Redux Toolkit 只维护需要跨页面复用的全局数据。
- `store` 需导出统一的 `RootState`、`AppDispatch` 和自定义 typed hooks，避免页面层手写类型。

## 8. 请求层设计

### 8.1 分层原则

接口调用分为三层：

1. 请求基础层：创建统一 `axios` 实例。
2. 业务服务层：在 `services` 中封装各业务域请求方法。
3. 页面调用层：页面只调用服务方法，不写裸请求。

### 8.2 请求实例职责

统一请求层负责以下职责：

- 注入 `baseURL`
- 注入 token
- 统一超时与请求头
- 处理业务成功码与错误码
- 处理登录过期
- 统一错误提示
- 封装分页请求常用参数

### 8.3 返回结构约定

首版默认约定后端返回 REST 风格 JSON。推荐结构如下：

```ts
interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
  traceId?: string;
}
```

分页接口推荐结构如下：

```ts
interface PageResult<T> {
  list: T[];
  pageNum: number;
  pageSize: number;
  total: number;
}
```

处理约定如下：

- `code === 0` 视为成功。
- 非成功状态统一进入错误处理。
- 认证失效码统一触发登录失效流程。
- 页面只消费 `data`，不在每个页面重复处理 `code` 和 `message`。

### 8.4 服务层组织方式

建议按业务域拆分：

```text
services/
  auth.service.ts
  dashboard.service.ts
  content.service.ts
  operation.service.ts
  user.service.ts
  role.service.ts
  menu.service.ts
```

规范要求如下：

- 一个文件负责一个业务域。
- 方法名使用“业务动作 + 资源名”的风格，例如 `fetchContentList`、`createActivity`。
- 页面不得直接拼接 URL 和处理重复请求逻辑。
- 服务层方法需要显式声明请求参数类型和返回值类型。

## 9. Mock 策略

### 9.1 总体策略

采用“开发环境可走 Mock，联调环境切真实接口”的策略，确保页面开发与后端开发解耦。

### 9.2 Mock 原则

- Mock 返回结构必须与未来真实接口保持一致。
- 页面层调用方式在 Mock 与真实接口之间保持不变。
- Mock 数据覆盖登录、权限、列表、详情、保存、删除、状态切换等主路径。
- Mock 方案优先服务开发效率，不在页面层加入与 Mock 强耦合的判断。

### 9.3 切换约定

通过环境变量控制是否启用 Mock，例如：

```env
VITE_USE_MOCK=true
VITE_API_BASE_URL=/api
```

当 `VITE_USE_MOCK=true` 时，请求走本地模拟；关闭后走真实后端接口。页面代码不感知切换细节。

## 12. 公共组件与复用策略

### 12.1 复用原则

- 优先复用现有公共组件，不重复封装 Ant Design 已经成熟的基础能力。
- 只有在多个业务页面重复出现、且存在统一交互规则时，才提取为公共组件。
- 公共组件保持职责单一，不承载过重业务语义。

### 12.2 建议沉淀的公共能力

- 页面容器组件

- 分页封装组件

## 13. 代码规范与协作约定

### 13.1 命名规范

- 页面组件使用 PascalCase。
- hooks 使用 `use` 前缀。
- store slice 使用业务域命名。
- 服务层方法使用动词开头。
- 权限码使用冒号分隔语义，例如 `content:list:view`。
- 类型命名使用 PascalCase，接口类型建议以业务语义命名，不使用无意义前缀。

### 13.2 文件拆分原则

- 页面文件保持聚焦，不把大量接口与工具函数堆在页面内。
- 通用逻辑优先放入 `features` 或 `utils`。
- 业务接口统一收敛到 `services`。
- 公共 UI 组件统一放入 `components`。
- `tsx` 用于页面与组件，`ts` 用于 store、services、types、utils 等非视图模块。

### 13.3 协作开发原则

- 先搭建路由骨架和页面骨架，再接入业务逻辑。
- 先基于 Mock 完成主要流程，再切真实接口联调。
- 新增公共组件前先判断是否可以通过组合 Ant Design 直接完成。
- 不允许在多个页面各自维护一份权限判断逻辑，必须统一封装。

## 15. 里程碑与实施顺序

## 16. 对外接口与关键约束

### 16.1 路由配置接口

每个业务路由必须至少包含：

- `path`
- `element`
- `meta.title`
- `meta.authCode`
- `meta.hidden`
- `meta.icon`
- `meta.keepAlive`

并在类型层统一收敛为 `AppRouteItem` 与 `AppRouteMeta`。

### 16.2 权限数据接口

权限数据至少包含：

- 用户基础信息
- 角色列表
- 菜单权限列表
- 按钮权限码列表

前端基于这些数据完成菜单、页面和按钮的三层权限控制。
权限返回值需在类型层收敛为 `PermissionPayload`。

### 16.3 请求返回接口

统一响应结构包含：

- `code`
- `message`
- `data`
- `traceId`

分页结果的 `data` 内固定包含：

- `list`
- `pageNum`
- `pageSize`
- `total`

请求层需通过 `ApiResponse<T>` 与 `PageResult<T>` 统一接口返回类型。

### 16.4 Store 边界

全局状态域至少包含：

- `auth`
- `permission`
- `app`

不得把全部页面局部状态都塞进全局 store。
项目需导出统一的 typed selector 和 dispatch hooks。

### 16.5 Mock 切换接口

Mock 与真实接口的切换只允许发生在请求层和 Mock 配置层，不允许业务页面通过条件判断分别调用两套路由。

## 17. 测试与验收场景

后续实现完成后，至少验证以下场景：

- 登录成功、登录失败、token 过期、退出登录
- 不同角色登录后的菜单、页面、按钮权限差异
- 内容列表页搜索、分页、状态切换、删除、批量操作
- 活动列表页搜索、分页、状态切换、删除、批量操作
- 内容新增/编辑的校验、保存、取消与异常提示
- 活动新增/编辑的校验、保存、取消与异常提示
- Mock 阶段与真实接口阶段的调用路径一致
- 首屏加载、路由跳转、异常页展示和错误提示可用
- 生产构建通过，环境变量切换不影响应用基础运行

## 18. 风险与后续演进

### 18.1 当前风险

- 首版引入 TypeScript 后，前期类型建模和基础模板搭建成本会略高于纯 JavaScript。
- 若业务需求频繁变化，过早细化类型定义可能带来一定重构成本，需要保持类型设计克制。
- 首版优先以通用后台形态落地，部分业务字段与流程需在实际需求明确后补齐。
- 过早封装复杂公共组件会增加维护成本，因此首版要克制抽象。

### 18.2 后续演进方向

- 根据业务稳定度逐步补齐更细的领域模型类型与接口声明。
- 根据模块数量增长情况评估是否引入更细的业务分包策略。
- 根据 SEO 或服务端需求评估是否迁移到更强服务端框架。
- 根据设计系统成熟度再评估是否替换或深度封装 Ant Design。

## 19. 实施默认假设

- 文档使用中文编写，采用 Markdown 格式维护。
- 目标用户为内部内容与运营团队，优先保障桌面端体验。
- 后端默认提供 REST 风格 JSON 接口，并支持基于 token 的认证。
- 首版默认使用 TypeScript，并要求核心公共结构具备明确类型定义。
- 首版不引入 SSR、微前端和复杂图表平台能力。
